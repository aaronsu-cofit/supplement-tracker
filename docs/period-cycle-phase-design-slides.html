生理週期 Phase Journey
設計討論 使用者主動回報、Phase-driven 每日推送、不做預測。
給操作 HQ 的同事 + 想理解流程的同事都看得懂。

tl;dr

一句話設計

使用者說「月經來了」/「結束了」 → 系統自動切 phase → 每天根據「user 在這個 phase 的第幾天」推對應內容。

不做預測排程、不做時間 fallback、不做循環計數、不做退出流程。
把現有的 UserAttribute / Journey / ContentItem / IntentRule 拼起來就成立。

⭕ 全員
decisions

5 個關鍵決定 → 5 個對應的「不做」

決定對應簡化
只靠 user 主動回報 不寫預測排程、onboarding 不問 last_period_date / cycle_length
Phase 內每天不同 + 準備很多天份量 不寫條件分支；超過準備天數就停推
Phase 沒切走放著 不寫 timeout cron；user 不更新就停在原 phase
跨 cycle 沒差別 不寫 cycle counter；每次切回經期就是新一輪
直接退出 不寫 graceful exit；user 想停就 unfollow / mute

⭕ 全員 🛠 HQ 操作
trigger

3 條觸發路徑（Intent Rule）

關鍵字（user 說）動作結果
「月經來了」「來了」「mc來」 set_attribute period_state=menstrual 切到 🩸 經期 phase
「月經結束」「結束了」「乾淨了」 set_attribute period_state=follicular 切到 🌱 濾泡 phase
（依後續設計） set_attribute period_state=luteal 切到 🍂 黃體 phase

⚠ 「進入黃體期」目前難以靠 user 主動辨識（沒有明顯生理訊號）。可能需要：postback 按鈕、或濾泡 Day 14 自動切、或乾脆不分濾泡/黃體只用兩 phase。下次討論再決定。

🛠 HQ 操作
hq setup

Journey「period_cycle」設定

HQ → 產品 → Journey → 新增：

{
  "key": "period_cycle",
  "name": "生理週期",
  "phases": [
    { "key": "menstrual",  "name": "🩸 經期" },
    { "key": "follicular", "name": "🌱 濾泡期" },
    { "key": "luteal",     "name": "🍂 黃體期" }
  ],
  "transitions": [
    { "to_phase": "menstrual",  "trigger": { "type": "attribute_equals", "attribute_key": "period_state", "value": "menstrual" } },
    { "to_phase": "follicular", "trigger": { "type": "attribute_equals", "attribute_key": "period_state", "value": "follicular" } },
    { "to_phase": "luteal",     "trigger": { "type": "attribute_equals", "attribute_key": "period_state", "value": "luteal" } }
  ]
}

不指定 from_phase = 從任何 phase 都可切過去（含黃體 → 經期循環自然成立）。

⭕ 全員 ⚙️ 程式
on transition

切 Phase 當下：立刻推 day_1

User 不用等到隔天才看到回應。在 evaluateJourneys() 切完 phase 後加一個 hook：

// in evaluateJourneys()
await upsertUserJourneyPhase(productId, userId, tpl.key, t.to_phase);

// 新增：phase 進入時立刻推 day_1 內容
const contentKey = `${t.to_phase}_day_1`;
await pushContentToUser(productId, userId, contentKey,
  'journey_phase_enter', `${tpl.key}:${t.to_phase}:day_1`);

若該 ContentItem 不存在 → 安靜略過（沒 day_1 也能用，例如某 phase 只想靠 daily cron 推）

⭕ 全員 ⚙️ 程式
daily cron

每日推送邏輯

// 每天早上 9 點跑（或自訂時間）
async function runPhaseDailyPush() {
  const rows = await db.userJourneyPhase.findMany({ where: {} });

  for (const row of rows) {
    const daysIn = daysBetween(today, row.entered_at) + 1;

    // 切 phase 當天已推 day_1，這裡跳過避免重複
    if (daysIn === 1) continue;

    const contentKey = `${row.phase_key}_day_${daysIn}`;
    const sourceRef = `${row.journey_key}:${row.phase_key}:day_${daysIn}`;

    // pushContentToUser 內部已防重發（message_log 唯一性）
    await pushContentToUser(row.product_id, row.user_id, contentKey,
      'journey_phase_daily', sourceRef);
  }
}

超出準備天數 → ContentItem 不存在 → push 函式 noop（不會壞）

engineering

程式要寫的部分（合計 ~1 天）

① Phase 進入立刻推 day_1（半天）

  • 修改 lib/journey.ts 的 evaluateJourneys()
  • upsertUserJourneyPhase 後加 hook：拼 ${to_phase}_day_1 呼叫 pushContentToUser
  • ContentItem 不存在就 noop

② 每日 cron 推當天內容（半天）

  • 新檔 lib/phaseDailyPush.ts + cron 註冊在 index.ts

  • idempotency 靠 message_log 既有防重發

Schema 不動。完全用現有的 UserJourneyPhase / ContentItem / message_log。

⭕ 全員
out of scope

明確「不做」（避免後續誤會）

  • ❌ 預測排程 — 不問上次經期、週期長度；不算下次月經何時來
  • ❌ Timeout fallback — phase 待太久不主動切；user 不回就停在原 phase
  • ❌ Cycle counter — 不記第幾輪；每次切回 menstrual 都當新一輪一樣對待
  • ❌ Graceful exit — 沒「我要結束課程」按鈕；user 想停就 unfollow
  • ❌ 個人化內容 — 所有 user 的 menstrual_day_3 看到一樣的字
  • ❌ 排除規則 — 不過濾 PCOS / 更年期 / 服避孕藥使用者（行銷端決定）
  • ❌ 跟保健品 / 鄭醫師營養課的串接 — Phase 1 不做，第二版迭代再加

這些不是放棄，是明確延後。每一條都是日後可加的迭代點。

下次會議前需要決定的 3 件事

  1. 進入新 phase 立刻推 day_1 對嗎？還是寧可隔天 cron 才推？
    建議：立刻推（user 馬上看到回應 = 好 UX）
  2. 超出準備天數時的行為？
    a) 停推 b) loop 到最後一天
    建議：a 停推
  3. 怎麼切到「黃體期」？
    a) user 手動觸發（但 user 不知道自己進入黃體了）
    b) 濾泡 Day 14 自動切（time-based）
    c) 不分濾泡/黃體，只做 menstrual + non-menstrual 兩 phase
    建議：b 或 c，需團隊討論
